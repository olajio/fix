Short answer: **it's not currently broken, but it has the same latent bugs, and I'd update it for consistency and future-proofing.**

Here's why 53001 hasn't blown up the way 53002 did — and where it will if you don't fix it.

## Why 53001 works today

Two structural differences save it:

**1. It only uses `start_alert_time`, not `end_alert_time`.** The midnight-wrap disaster in 53002 was triggered by `end_alert_time = 19:00-05:00` normalizing to `00:00` UTC the next day. Since 53001 never touches `end_alert_time`, that specific failure mode can't fire.

**2. None of your current `start_alert_time` values wrap past UTC midnight.** All enabled sessions start at `05:02-05:00` (10:02 UTC) or `00:02-05:00` (05:02 UTC). Neither wraps. So `doc['start_alert_time'].value.getHour()` returns a sensible positive integer, and the `-1` summer adjustment stays positive too.

Trace-through for TRUMID at 5:02 AM EDT (Jun 30 summer):
```
paris_offset = 2 (CEST)                    → summer branch
alert_time   = doc.getHour() - 1 = 10 - 1 = 9
now.getHour() = 9 (UTC)                    → 9 == 9 ✓
doc.minute (2) <= now.minute (2)           ✓
doc.minute (2) > 2 - 5 = -3                ✓
return true
```

Verified working. And the same trace works for winter EST, and for the March/November transition weeks (via the `paris_offset == 1 && ny_time_diff == 4` branch). So the current configs are covered.

## Where 53001 will break the moment configs change

The bugs are still latent:

**Anyone adding a session with `start_alert_time` between 19:01-05:00 and 24:00-05:00** (i.e., an evening start in NY local time) will hit the exact same midnight-wrap issue 53002 has. `getHour()` returns 0 or wraps low, the summer `-1` produces `-1` or a small wrong number, alert never fires. This is one config change away.

**The `ny_time_diff` calculation is still broken at UTC midnight.** `nowUTC.getHour() - nowNY.getHour()` = `1 - 21` = `-20` at 01:00 UTC. The `else if` transition branch (Paris CET + NY EDT window) would silently be skipped. Not affecting anything right now because no configs have start times near UTC midnight during transition weeks, but fragile.

**The hour-comparison logic can't span an hour boundary.** If someone sets `start_alert_time` to something like `05:58` and `lookup_interval` stays at 5, the window `[05:58, 06:03)` crosses the hour boundary. At 06:00 the script's `now.getHour() == alert_time` check fails and it returns false — you'd lose 3 of the 5 minutes of detection.

**Same misleading comment, same brittle Paris-as-proxy dependency** — these will bite in some future EU DST rule change (they've been discussing abolition for years).

## Equivalent fix for 53001

Same design, adjusted for start-only-plus-lookup-interval semantics:

```painless
ZonedDateTime nowNY = ZonedDateTime.ofInstant(
    Instant.ofEpochMilli(new Date().getTime()),
    ZoneId.of('America/New_York')
);
int nowMin = nowNY.getHour() * 60 + nowNY.getMinute();

// Rebase to -05:00 to recover the original wall-clock hh:mm
def s = doc['start_alert_time'].value.withZoneSameInstant(ZoneId.of('-05:00'));
int startMin = s.getHour() * 60 + s.getMinute();

int lookup = {{ ctx.metadata.lookup_interval }};

// Fire if now is in [start, start + lookup) minutes-of-day, with midnight wrap
int diff = nowMin - startMin;
if (diff < 0) diff += 1440;
return diff >= 0 && diff < lookup;
```

Verification for TRUMID at 5:02 EDT:
- `nowNY = 05:02-04:00` → `nowMin = 302`
- `s` rebased to -05:00 = `05:02-05:00` → `startMin = 302`
- `diff = 0`, `0 >= 0 && 0 < 5` → **true** ✓

At 5:06 EDT: `diff = 4`, `< 5` → true. At 5:07 EDT: `diff = 5`, `< 5` → false. Same 5-minute window as before, but now it handles hour boundaries and arbitrary start times cleanly.

## My recommendation

**Yes, update 53001 too.** Not because it's actively broken — for your current five session configs it's fine — but because:

1. **Consistency.** All three watchers (53001, 53002_to_admin, 53002_from_admin) share this identical broken pattern. Fixing one and leaving the others is confusing for whoever inherits this.
2. **Future-proofing.** The next person to add a session with an evening start time will silently get no alerts and no signal that anything is wrong.
3. **The fix is drop-in.** Same shape as the 53002 fix, no schema changes, no data migration.

Bundle all three updates into one AWX deployment. Same PR, same rollback surface.
